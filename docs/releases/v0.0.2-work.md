# v0.0.2 тАФ Work breakdown

Implementation sub-problems for [v0.0.2](v0.0.2.md). Tackle roughly in order; later items depend on earlier ones.

## WP1 тАФ Redis infrastructure

- `manifests/local/redis.yaml` (Deployment + Service in `actordock` namespace)
- Add to `manifests/local/kustomization.yaml`
- `install-local.sh` applies Redis with Actordock stack
- Platform config: `ACTORDOCK_REDIS_ADDR` (or similar) env var; wire in `internal/config/platform.go`
- Platform Deployment waits for Redis reachable at startup

## WP2 тАФ Sandbox store + create/delete integration

- `internal/store` (or `internal/sandboxstore`): Redis-backed sandbox metadata
- Record fields: `sandbox_id`, `actor_id` (same id for v0.0.2), `template`, `created_at`, `status`
- `POST /sandboxes` writes Redis record after successful `CreateActor`
- `DELETE /sandboxes/{id}` removes Redis record after successful `DeleteActor`
- Unit tests with miniredis or similar

## WP3 тАФ Platform get and list APIs

- `GET /sandboxes/{id}` тАФ E2B-shaped JSON; 404 if not in Redis
- `GET /sandboxes` тАФ list non-terminated sandboxes from Redis
- On read: call runtime-api `GetActor` to sync status (`running` / `pending` / `failed`) from runtime `Actor.status`
- Map runtime statuses: e.g. `STATUS_RESUMING` тЖТ `pending`, `STATUS_RUNNING` тЖТ `running`, missing actor тЖТ remove record or 404
- `internal/runtimeapi`: add `GetActor` client method
- Platform unit tests for handlers

## WP4 тАФ E2E and documentation

- E2E: `Sandbox.get_info(sandbox_id)` after create тЖТ `running`
- E2E: `Sandbox.list()` includes active sandbox; after kill GET тЖТ 404
- E2E: two concurrent sandboxes appear in LIST
- Optional smoke: Platform pod restart, GET still works for live sandbox
- Update `docs/user/quickstart.md` (visibility section)
- Update `docs/releases/v0.0.2.md` Done when checkboxes when complete
