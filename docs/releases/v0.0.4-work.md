# v0.0.4 — Work breakdown

Implementation sub-problems for [v0.0.4](v0.0.4.md). Tackle roughly in order; later items depend on earlier ones.

## WP1 — Store `on_timeout` lifecycle field

- Add `OnTimeout` to `store.Sandbox` JSON record (e.g. `"kill"` | `"pause"`; only `"kill"` enforced in v0.0.4)
- Constant: `OnTimeoutKill = "kill"` (E2B `SandboxOnTimeout.kill`)
- Helper: default `OnTimeoutKill` when empty on create
- Unit tests in `internal/store`

**E2B semantics:** default lifecycle is `on_timeout=kill`. `pause` is stored for forward compat but **not** acted on until [v0.0.5](v0.0.5.md).

## WP2 — Platform create persists lifecycle

- `POST /sandboxes`: accept optional `lifecycle` object (`onTimeout` / snake_case per E2B JSON)
- If omitted: persist `on_timeout=kill`
- If `lifecycle.onTimeout` is `"pause"` or `autoPause=true`: **accept and store** (for v0.0.5) but scheduler in v0.0.4 only processes `kill` records — **or** reject non-`kill` with **400** (pick one; recommend **store + ignore pause** only if create body already sent by SDK tests; otherwise **400** until v0.0.5)
- Reject unknown `onTimeout` values with **400**
- Platform unit tests

No new HTTP routes.

## WP3 — List expired sandboxes

- `store` method: `ListExpired(ctx, now time.Time) ([]Sandbox, error)` (scan + filter `expires_at.Before(now)` and non-zero `expires_at`)
- Alternatively filter in scheduler from `List()`; prefer dedicated helper for tests
- Unit tests with miniredis (create + set_timeout shifts expiry)

## WP4 — Shared expiry kill path

- `internal/scheduler` (or shared package): `ExpireSandbox(ctx, sandboxID)`:
  1. Load record from Redis (skip if missing)
  2. Skip if `on_timeout != kill` (pause deferred to v0.0.5)
  3. Call runtime `DeleteSandbox` (suspend + delete — reuse `internal/runtimeapi.Client`)
  4. Delete Redis record
- **Idempotent:** actor already gone in runtime → still purge Redis; no panic on double-run
- Unit tests with fake store + fake runtime client

Align behavior with Platform `DELETE /sandboxes/{id}` where possible (extract shared helper if duplication appears).

## WP5 — Scheduler daemon

- `cmd/scheduler/main.go`
- `internal/config/scheduler.go`: `SCHEDULER_REDIS_ADDR`, `SCHEDULER_RUNTIME_API_ADDR`, `SCHEDULER_POLL_INTERVAL` (default e.g. 5s)
- Loop: `ListExpired` → `ExpireSandbox` per id; log success/failure per sandbox
- Graceful shutdown on SIGTERM
- `Makefile`: add `scheduler` to `BINARIES` / `build-images` (ko)
- `.ko.yaml` entry if needed

## WP6 — Scheduler deployment

- `manifests/local/scheduler.yaml`: Deployment **replicas: 1**, env from ConfigMap (redis, runtime-api, poll interval)
- Add to `manifests/local/kustomization.yaml`
- `hack/lib/common.sh` `deploy_actordock_images`: build + apply scheduler; optional rollout wait
- `install-local.sh` brings up scheduler with platform/router

## WP7 — E2E and documentation

- E2E: `Sandbox.create(..., timeout=30)` → sleep ~35s → `Sandbox.get_info` → **404**
- E2E: `set_timeout(120)` before expiry → sandbox still alive after original window
- E2E: manual `kill()` before expiry → GET **404**; scheduler does not resurrect
- Optional: kubectl / runtime-api check — no orphan actor after expiry
- Brief `docs/user/quickstart.md` note: timeout is enforced automatically (v0.0.4+)
- `hack/verify-local.sh` / CI `e2e` job runs new tests
- Update `docs/releases/v0.0.4.md` Done when checkboxes when complete

## E2B / SDK reference (this version)

| Behavior | Notes |
|----------|--------|
| `Sandbox.create(timeout=30)` | After ~30s, sandbox removed without `kill()` |
| `sandbox.set_timeout(n)` | Resets expiry; scheduler respects new `expires_at` |
| `sandbox.kill()` | Unchanged; works before scheduler fires |
| Default `on_timeout` | `kill` (E2B default) |

No new Platform REST routes; scheduler is internal.
