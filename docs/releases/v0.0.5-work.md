# v0.0.5 — Work breakdown

Implementation sub-problems for [v0.0.5](v0.0.5.md). Tackle roughly in order; later items depend on earlier ones.

**Full-field rule:** each shipped operation includes **all** OpenAPI request/response fields. See [e2b-rest-parity.md](e2b-rest-parity.md).

## WP1 — Store auto_resume and accept pause on create

- Add `AutoResume` (bool) to `store.Sandbox` JSON record (E2B `SandboxLifecycle.autoResume`)
- Platform: accept `lifecycle.onTimeout=pause`, `autoPause=true`, full `autoResume` config on `POST /sandboxes` (remove v0.0.4 reject)
- Persist `on_timeout=pause` + `auto_resume` on create
- Unit tests in `internal/store` and `internal/platform`

**E2B semantics:** `autoPause=true` ⇔ `on_timeout=pause`. `auto_resume` only valid when `on_timeout=pause`.

## WP2 — Substrate SuspendActor client

- `internal/substrate`: `SuspendSandbox(ctx, actorID)` wrapping `SuspendActor` gRPC
- Map `ErrNotFound`, idempotent when already suspended
- Unit tests with fake gRPC or integration-style mocks

Used by scheduler pause path and Platform `POST …/pause`.

## WP3 — Scheduler expiry suspend path

- Extend `internal/scheduler` `ExpireSandbox` (or sibling `SuspendSandbox`):
  1. Load record from Redis
  2. If `on_timeout=pause`: call `SuspendSandbox`; set store `status=paused`; **do not** delete Redis
  3. If `on_timeout=kill`: existing purge path unchanged
- Unit tests: pause sandbox survives expiry; kill path unchanged

## WP4 — Platform pause/resume REST + GET lifecycle fields

- `POST /sandboxes/{id}/pause` → **204** (no body)
- `POST /sandboxes/{id}/resume`: full `ResumedSandbox` request (`timeout`, `autoPause`); response full `Sandbox` (8 fields)
- `GET /sandboxes/{id}` and list items: return full `SandboxLifecycle` (`onTimeout`, `autoResume`); `state=paused` when suspended
- `storeStatusFromActor` / `ActorStateE2B` already map Substrate suspended → `paused`
- Platform unit tests per route + field assertions

## WP5 — Router auto-resume after scheduler pause

- Verify `Router` proxy path resumes paused/suspended actors on first command (existing `ResumeSandboxBackend`)
- Fix gaps if command fails after scheduler suspend (e.g. stale store status, missing resume)
- Router unit test: paused actor → proxy → backend assigned

## WP6 — E2E and documentation

- E2E: `Sandbox.create(..., lifecycle={"on_timeout":"pause","auto_resume":True}, timeout=60)` → sleep past timeout → `get_info` shows `paused` (not 404)
- E2E: `commands.run` after pause succeeds without explicit `resume` API call
- E2E: explicit `POST …/pause` and `POST …/resume` (SDK or HTTP)
- E2E: `on_timeout=kill` sandboxes still auto-deleted (v0.0.4 regression)
- Brief `docs/user/quickstart.md` note on pause lifecycle
- Update `docs/releases/v0.0.5.md` Done when checkboxes; parity tracker `partial` → `done` for lifecycle fields

## E2B / SDK reference (this version)

| SDK / HTTP | Notes |
|------------|--------|
| `Sandbox.create(lifecycle={"on_timeout":"pause","auto_resume":True})` | Survives timeout; scheduler suspends actor |
| `sandbox.commands.run(...)` after pause | Router resumes; no manual `resume()` required when `auto_resume=True` |
| `sandbox.pause()` / `sandbox.resume()` | Maps to `POST …/pause`, `POST …/resume` |
| Default `on_timeout=kill` | Unchanged from v0.0.4 |
