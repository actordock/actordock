# v0.0.9 — Work breakdown

Implementation sub-problems for [v0.0.9](v0.0.9.md). Four new Platform REST operations plus **runtime volume bind-mount** into sandboxes (not metadata-only stubs).

**Full-field rule:** each shipped operation includes **all** OpenAPI request/response fields. See [e2b-rest-parity.md](e2b-rest-parity.md).

## WP1 — Volume Redis store

- `internal/store/volume.go`: `VolumeID`, `Name`, `Token`, `HostPath`, `CreatedAt`
- CRUD: `PutVolume`, `GetVolume`, `ListVolumes`, `DeleteVolume`, `GetVolumeByName`
- Token: cryptographically random (e.g. 32 bytes, base64url); stored hashed or plaintext per E2B semantics (return on create/get only)
- Name uniqueness enforced at store layer
- Host path: `{VolumeRoot}/{volumeID}/` (see WP5)
- Unit tests: round-trip, duplicate name, not-found

## WP2 — Platform `POST /volumes` + `GET /volumes`

- `POST /volumes` — full `NewVolume` schema (`name` pattern `^[a-zA-Z0-9_-]+$`)
  - Generate `volumeID`, token, allocate host dir (mkdir via WP5 helper or defer to first mount)
  - **201** `VolumeAndToken` (`volumeID`, `name`, `token`)
  - Duplicate name → **400**; invalid name → **400**
- `GET /volumes` — **200** `[Volume]` (`volumeID`, `name` only — no token in list)
- Platform unit tests with fake store

## WP3 — Platform `GET /volumes/{id}` + `DELETE /volumes/{id}`

- `GET /volumes/{volumeID}` — **200** `VolumeAndToken`; missing → **404**
- `DELETE /volumes/{volumeID}` — **204**; missing → **404**
- Optional: reject delete if volume referenced by a running sandbox (stretch; document behavior)
- Platform unit tests

## WP4 — Sandbox `volumeMounts` accept + round-trip

- `POST /sandboxes`: accept `volumeMounts: [{name, path}]`
  - Validate each `name` resolves to an existing volume (by name or volumeID)
  - Validate `path` is absolute, no `..`, unique per sandbox
  - Persist on `store.Sandbox.VolumeMounts`
- `GET /sandboxes/{id}`: return stored `volumeMounts` in `SandboxDetail`
- `GET /sandboxes`, `GET /v2/sandboxes`: include `volumeMounts` on listed items
- Unknown volume name → **400** on create
- Platform unit tests assert JSON keys present

## WP5 — Runtime bind-mount (Substrate + actordock)

Real persistence: files written inside the sandbox at the mount path survive sandbox kill and are visible in a new sandbox mounting the same volume.

### WP5a — Substrate: per-actor bind mounts

- Extend `CreateActorRequest` (and Actor persisted state) with `repeated BindMount { host_path, container_path, read_only }`
- atecontroller forwards mounts to atelet `RunRequest` on first boot and re-attaches on `ResumeActor`
- atelet `buildActorOCISpec`: add rw bind mounts (same pattern as `IdentityMountPath`); `createMountPoint` for each container path
- Unit tests in substrate: OCI spec contains expected mount entries
- **Note:** golden-snapshot actors boot once; per-sandbox actors are non-golden — mounts apply at actor Run

### WP5b — actordock: volume root + create wiring

- Config: `VolumeRoot` (default `/var/lib/actordock/volumes` on worker host)
- WorkerPool / atelet manifest: hostPath volume exposing `VolumeRoot` to workers (Kind local dev)
- On `POST /volumes`: `MkdirAll` `{VolumeRoot}/{volumeID}` on worker — Platform calls a small init or relies on atelet lazy-create on first Run (pick one, document)
- Extend `substrate.Client.CreateAndResumeSandbox` (or new param) to pass resolved bind mounts:
  - `{HostPath: volume.HostPath, ContainerPath: mount.Path}`
- `handleCreateSandbox`: after volume validation, call Substrate with mounts **before** envd ready wait
- Unit tests with fake actors asserting mount list passed through

### WP5 acceptance

- E2B demo path works:
  1. `Volume.create("my-vol")`
  2. `Sandbox.create(volume_mounts={"/mnt/data": "my-vol"})`
  3. `commands.run("echo hello > /mnt/data/file.txt")`
  4. Kill sandbox
  5. New sandbox, same mount → `cat /mnt/data/file.txt` → `hello`
