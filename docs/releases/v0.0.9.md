# v0.0.9

**Status:** released (2026-06-21)

## Target

**Volumes** — E2B volume CRUD on Platform plus `volumeMounts` metadata on sandboxes.

| Before | After |
|--------|-------|
| No volume API | Create/list/get/delete volumes via REST |
| `volumeMounts` ignored | Validated + persisted on create; returned on get/list |

**Note:** Runtime bind-mount into sandboxes is **backlog** ([#48](https://github.com/actordock/actordock/issues/48) / [#49](https://github.com/actordock/actordock/issues/49)), not v0.0.9.

## Goal

```python
from e2b import Volume, Sandbox

volume = Volume.create("my-data")
sbx = Sandbox.create(template="base", volume_mounts={"/mnt/data": volume})
info = httpx.get(f"{API}/sandboxes/{sbx.sandbox_id}", headers=headers).json()
assert info["volumeMounts"] == [{"name": "my-data", "path": "/mnt/data"}]
```

## In scope

| Area | Deliverable |
|------|-------------|
| Platform | Full volume CRUD: `GET/POST /volumes`, `GET/DELETE /volumes/{id}` |
| Store | Volume metadata + token + host path in Redis |
| Sandbox API | Accept `volumeMounts` on create; return on get/list |
| Tests | Unit CRUD + E2E volume CRUD + `volumeMounts` round-trip |

## Out of scope (backlog)

- Substrate per-actor bind mounts ([#48](https://github.com/actordock/actordock/issues/48))
- Worker hostPath wiring + cross-sandbox file persistence ([#49](https://github.com/actordock/actordock/issues/49))
- GCS/S3 backing
- Standalone volume file SDK when not mounted

## API

| Method | Path |
|--------|------|
