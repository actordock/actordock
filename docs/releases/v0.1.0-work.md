# v0.1.0 — Work breakdown

Implementation sub-problems for [v0.1.0](v0.1.0.md). **Pilot + 60% E2B REST** — filesystem + secure envd, sandbox schema backfill, six new Platform routes, Helm deploy.

**Full-field rule:** each shipped operation includes **all** OpenAPI request/response fields. See [e2b-rest-parity.md](e2b-rest-parity.md).

**Self-hosted defaults:** single logical team, dev-style API keys in config/Redis unless noted. No Supabase / multi-tenant billing.

---

## WP1 — envd filesystem (read, write, list)

SDK demo path: `sandbox.files.write` / `read` / `list` (E2B uses HTTP `GET|POST|DELETE /files` plus Connect RPC for `ListDir`, `Stat`, …).

### Proto + codegen

- Vendor or copy [E2B `filesystem.proto`](https://github.com/e2b-dev/E2B/blob/main/spec/envd/filesystem/filesystem.proto) into `spec/envd/filesystem/` (or `pkg/envd/filesystem/`)
- `hack/generate-proto.sh` → `pkg/envd/filesystem/` + connect handlers
- Register on envd mux alongside `Process` and `/logs`

### HTTP `/files` (v0.1.0 minimum)

Match E2B envd API used by SDK `Filesystem.read` / `write` / `remove`:

- `GET /files` — query `path`; response body = file bytes (or JSON error)
- `POST /files` — multipart or raw body per E2B SDK (`to_upload_body`); create/overwrite
- `DELETE /files` — optional for round-trip tests if SDK `remove` needed
- Auth: accept `X-Access-Token` when `secure=true` (stub OK until WP2 wires tokens)

### Connect RPC (v0.1.0 minimum)

- `ListDir` — required for `files.list` / directory listing in SDK
- `Stat` — support path checks
- Defer to post-v0.1.0 unless SDK demo needs them: `WatchDir`, `Move`, `MakeDir`, `Remove` RPC (HTTP delete may suffice)

### Tests

- `internal/envd/files_*_test.go`: write → read round-trip on temp dir; list returns entries
- Manual: `Sandbox.create(secure=False)` file ops before WP2; with WP2: `secure=True`

## WP2 — Secure sandbox + envd auth

Today `POST /sandboxes` with `"secure": true` → **400**. v0.1.0 enables the full path.

### Platform

- On `secure=true` create: generate `envd_access_token` (and `traffic_access_token` if required by OpenAPI `Sandbox` response); persist on `store.Sandbox`
- Return all 8 `Sandbox` response fields including `envdAccessToken`, `domain`
- `GET /sandboxes/{id}`: return `envdAccessToken` when caller owns sandbox (match E2B visibility rules; self-hosted: always return for API-key auth)
- `POST …/resume`: populate `envdAccessToken` / `trafficAccessToken` on `ResumedSandbox` response (parity row)
- Reject or no-op fields documented in OpenAPI (`mcp`, etc.) with stored defaults

### Router

- Forward `X-Access-Token` (and any E2B auth headers SDK sends) to envd on command/file/process RPCs
- Connect path unchanged except auth header pass-through

### envd

- Validate `X-Access-Token` against token issued at sandbox start when secure mode enabled
- Dev escape hatch: `E2B_VALIDATE_API_KEY=false` / insecure create still works for Kind (document env var)

### Tests

- Platform: create `secure=true` → non-empty `envdAccessToken`; get sandbox returns token
- Router: request without token → 401 on envd when secure
- E2E (WP7): `Sandbox.create(secure=True)` + `commands.run` + `files.write` without `E2B_VALIDATE_API_KEY=false`

## WP4 — Auth REST: API keys and access tokens

Four new operations (ops #47–50). Self-hosted, single-tenant: no `GET /teams` or other multi-team routes in v0.1.0.

### `GET /api-keys`, `POST /api-keys`

- List keys from config file or Redis (v0.1.0: at minimum the configured `ACTORDOCK_API_KEY` as one `TeamAPIKey` entry)
- `POST` creates a new random key, stores hash, returns `CreatedTeamAPIKey` with **full** schema including `key` once
- New keys validate on `X-API-KEY` alongside the configured bootstrap key
- Defer `PATCH`/`DELETE` per-key (#51–52) — out of v0.1.0 scope

### `POST /access-tokens`, `DELETE /access-tokens/{id}`

- Issue tokens for dashboard/CLI; separate Redis namespace (not sandbox `envdAccessToken`)
- **Full** `NewAccessToken` / `CreatedAccessToken` schemas
- Defer Bearer-auth wiring for all Platform routes — CRUD + persistence only in v0.1.0

### Tests

- Platform httptest per route; field presence assertions
- E2E smoke: `GET /api-keys` with `X-API-KEY`

## WP5 — Template `POST` + `PATCH`

Ops #26 and #30. **No build pipeline** — metadata/catalog only (same semantic gap as v0.0.10).

### `POST /templates`

- Accept `NewTemplate` body with **all** fields
- v0.1.0 behavior: register template metadata in catalog store **or** create/update `ActorTemplate` CR if runtime supports minimal create without build ([#59](https://github.com/actordock/actordock/issues/59) backlog if not)
- **201** `Template` with full 16-field response (synthetic `buildID`, `buildStatus=ready` if no build)

### `PATCH /templates/{templateID}`

- Accept full `TemplateUpdateRequest`; return `TemplateUpdateResponse` with all fields
- Unknown template → **404**

### Tests

- Platform unit tests; optional SDK `Template` calls if exposed
